A knowledge graph's power grows significantly when you connect it to the wider web of linked data. ANC Lab Knowledge Hub provides four integration pathways: federating queries against remote SPARQL endpoints, importing RDF dumps from public datasets, linking your entities to external identifiers using `owl:sameAs`, and mapping structured tabular or JSON data to RDF through the built-in import endpoint. This guide walks through each method and shows you how to build a coherent external-data integration workflow.

<Warning>
Before importing any external dataset, verify its license. Common licenses impose restrictions that affect your usage:
- **CC BY / CC BY-SA** — attribution required; SA requires your derived work to carry the same license.
- **CC0 / Public Domain** — no restrictions.
- **ODC-ODbL** (used by OpenStreetMap and some Wikidata exports) — share-alike for databases.
- Some DBpedia and Wikidata content mixes licenses at the triple level.

When in doubt, isolate externally-sourced data in a dedicated named graph so you can identify and remove it if needed.
</Warning>

## Importing from a Remote SPARQL Endpoint

The platform can query a remote SPARQL endpoint and write the results directly into your graph. This is useful for pulling a specific slice of DBpedia, Wikidata, or any other publicly accessible SPARQL endpoint.

The workflow is: run a `CONSTRUCT` query against the remote endpoint to produce an RDF result, save it as a Turtle file, then upload it to `POST /graphs/{id}/import`. The example below shows both steps.

<CodeGroup>

```bash curl
# Step 1: Fetch data from a remote SPARQL endpoint and save as Turtle
curl -G "https://dbpedia.org/sparql" \
  --data-urlencode "query=PREFIX dbo: <http://dbpedia.org/ontology/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
CONSTRUCT { ?company a dbo:Company ; rdfs:label ?label . }
WHERE { ?company a dbo:Company ; rdfs:label ?label .
        FILTER(LANG(?label) = \"en\") } LIMIT 500" \
  -H "Accept: text/turtle" \
  -o dbpedia-companies.ttl

# Step 2: Import the Turtle file into your graph
curl -X POST https://api.anclab.io/v1/graphs/grph_3mNpQrStuv/import \
  -H "Authorization: Bearer anchub_sk_..." \
  -F "file=@dbpedia-companies.ttl;type=text/turtle" \
  -F "namedGraph=https://data.anclab.io/ecommerce/imports/dbpedia-companies" \
  -F "validate=false"
```

```python Python
import requests

BASE_URL = "https://api.anclab.io/v1"
GRAPH_ID  = "grph_3mNpQrStuv"
HEADERS   = {
    "Authorization": "Bearer anchub_sk_...",
    "Content-Type": "application/json",
}

# Step 1: Fetch data from a remote SPARQL endpoint
construct_query = """
PREFIX dbo:  <http://dbpedia.org/ontology/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

CONSTRUCT {
  ?company a dbo:Company ;
           rdfs:label ?label .
}
WHERE {
  ?company a dbo:Company ;
           rdfs:label ?label .
  FILTER(LANG(?label) = "en")
}
LIMIT 500
"""

remote_response = requests.get(
    "https://dbpedia.org/sparql",
    params={"query": construct_query},
    headers={
        "Accept": "text/turtle",
        "User-Agent": "ANCLabKnowledgeHub/1.0",
    },
    timeout=60,
)
remote_response.raise_for_status()
turtle_data = remote_response.content

# Step 2: Import the fetched RDF into your graph
r = requests.post(
    f"{BASE_URL}/graphs/{GRAPH_ID}/import",
    headers={"Authorization": "Bearer anchub_sk_..."},
    files={"file": ("dbpedia-companies.ttl", turtle_data, "text/turtle")},
    data={
        "namedGraph": "https://data.anclab.io/ecommerce/imports/dbpedia-companies",
        "validate": "false",
    },
)
job = r.json()
print(f"Import job: {job['jobId']}  status: {job['status']}")
```

</CodeGroup>

<Note>
Remote SPARQL endpoints are subject to rate limits — DBpedia allows roughly 20 requests per minute per IP, and Wikidata's Query Service enforces a 60-second timeout and blocks clients that send more than 5 concurrent requests. If you need large volumes of data, download a full RDF dump (see the next section) instead of querying the live endpoint.
</Note>

Public linked data sources you can federate against or import from:

| Source | Endpoint / Dump URL | License |
|--------|--------------------|---------| 
| **DBpedia** | `https://dbpedia.org/sparql` | CC BY-SA 3.0 |
| **Wikidata** | `https://query.wikidata.org/sparql` | CC0 |
| **schema.org** | `https://schema.org/version/latest/schemaorg-current-https.ttl` | CC BY-SA 4.0 |

## Importing RDF Dumps

For bulk data, download an RDF dump and upload it directly to the platform. The import endpoint accepts Turtle, N-Triples, N-Quads, RDF/XML, and JSON-LD files up to 2 GB.

<CodeGroup>

```bash curl
# Upload a pre-downloaded DBpedia product dump
curl -X POST https://api.anclab.io/v1/graphs/grph_3mNpQrStuv/import \
  -H "Authorization: Bearer anchub_sk_..." \
  -F "file=@dbpedia-products-en.ttl;type=text/turtle" \
  -F "namedGraph=https://data.anclab.io/ecommerce/imports/dbpedia-products" \
  -F "validate=false" \
  -F "batchSize=10000"
```

```python Python
def import_rdf_dump(file_path: str, named_graph: str, mime_type: str = "text/turtle") -> dict:
    with open(file_path, "rb") as f:
        r = requests.post(
            f"{BASE_URL}/graphs/{GRAPH_ID}/import",
            headers={"Authorization": "Bearer anchub_sk_..."},
            files={"file": (file_path, f, mime_type)},
            data={
                "namedGraph": named_graph,
                "validate": "false",   # skip SHACL for bulk raw import
                "batchSize": "10000",
            },
        )
    r.raise_for_status()
    return r.json()

job = import_rdf_dump(
    "dbpedia-products-en.ttl",
    "https://data.anclab.io/ecommerce/imports/dbpedia-products",
)
print(f"Import job: {job['jobId']}")


def wait_for_job(job_id: str, poll_interval: int = 5) -> dict:
    import time
    while True:
        status = requests.get(
            f"{BASE_URL}/jobs/{job_id}",
            headers={"Authorization": "Bearer anchub_sk_..."},
        ).json()
        print(f"  [{status['status']}] {status.get('progress', 0)}% complete")
        if status["status"] in ("completed", "failed"):
            return status
        time.sleep(poll_interval)

result = wait_for_job(job["jobId"])
print(f"Done: {result['triplesImported']} triples imported")
```

</CodeGroup>

## Entity Linking with owl:sameAs

Entity linking connects your local entities to authoritative external identifiers. Once you declare `ec:supplier/acme-corp owl:sameAs dbpedia:ACME_Corporation`, reasoners can merge properties from both descriptions automatically.

<Steps>

### Identify Match Candidates

Run a SPARQL query to find your suppliers' names, then search external datasets for matching labels.

```python Python
# Step 1: Pull your supplier names
supplier_query = """
PREFIX ec: <https://ontologies.anclab.io/ecommerce/v1#>

SELECT ?supplier ?name
WHERE {
  ?supplier a ec:Supplier ;
            ec:hasName ?name .
}
"""

r = requests.post(
    f"{BASE_URL}/graphs/{GRAPH_ID}/query",
    data=supplier_query,
    headers={**HEADERS, "Content-Type": "application/sparql-query"},
)
suppliers = r.json()["results"]["bindings"]
print(f"Found {len(suppliers)} suppliers to link")
```

### Search an External Endpoint for Matches

Query Wikidata for organisations with matching labels using its SPARQL endpoint.

```python Python
import urllib.parse

WIKIDATA_ENDPOINT = "https://query.wikidata.org/sparql"

def find_wikidata_match(label: str) -> list[dict]:
    query = f"""
    SELECT ?item ?itemLabel WHERE {{
      ?item wdt:P31 wd:Q4830453 .
      ?item rdfs:label "{label}"@en .
      SERVICE wikibase:label {{ bd:serviceParam wikibase:language "en". }}
    }}
    LIMIT 3
    """
    r = requests.get(
        WIKIDATA_ENDPOINT,
        params={"query": query, "format": "json"},
        headers={"User-Agent": "ANCLabKnowledgeHub/1.0"},
        timeout=30,
    )
    return r.json()["results"]["bindings"]
```

### Assert owl:sameAs Links

Post the confirmed `owl:sameAs` triples to your graph using `POST /graphs/{id}/triples`.

<CodeGroup>

```bash curl
curl -X POST https://api.anclab.io/v1/graphs/grph_3mNpQrStuv/triples \
  -H "Authorization: Bearer anchub_sk_..." \
  -H "Content-Type: application/json" \
  -d '{
    "triples": [
      {
        "subject":   "https://data.anclab.io/ecommerce/supplier/acme-corp",
        "predicate": "http://www.w3.org/2002/07/owl#sameAs",
        "object":    "http://www.wikidata.org/entity/Q12345678"
      },
      {
        "subject":   "https://data.anclab.io/ecommerce/supplier/acme-corp",
        "predicate": "http://www.w3.org/2002/07/owl#sameAs",
        "object":    "http://dbpedia.org/resource/ACME_Corporation"
      }
    ]
  }'
```

```python Python
same_as_triples = [
    {
        "subject":   "https://data.anclab.io/ecommerce/supplier/acme-corp",
        "predicate": "http://www.w3.org/2002/07/owl#sameAs",
        "object":    "http://www.wikidata.org/entity/Q12345678",
    },
    {
        "subject":   "https://data.anclab.io/ecommerce/supplier/acme-corp",
        "predicate": "http://www.w3.org/2002/07/owl#sameAs",
        "object":    "http://dbpedia.org/resource/ACME_Corporation",
    },
]

r = requests.post(
    f"{BASE_URL}/graphs/{GRAPH_ID}/triples",
    json={"triples": same_as_triples},
    headers=HEADERS,
)
print(f"Inserted {r.json()['inserted']} sameAs triple(s)")
```

</CodeGroup>

### Trigger Entity Merging

Ask the platform's reasoner to materialise inferred triples from `owl:sameAs` links so you can query both local and external properties transparently.

<CodeGroup>

```bash curl
curl -X POST https://api.anclab.io/v1/graphs/grph_3mNpQrStuv/infer \
  -H "Authorization: Bearer anchub_sk_..." \
  -H "Content-Type: application/json" \
  -d '{ "rules": ["sameAs-propagation", "subClassOf"], "persist": true }'
```

```python Python
r = requests.post(
    f"{BASE_URL}/graphs/{GRAPH_ID}/infer",
    json={"rules": ["sameAs-propagation", "subClassOf"], "persist": True},
    headers=HEADERS,
)
infer_result = r.json()
print(f"Inference complete: {infer_result['triplesAdded']} new triples materialised")
```

</CodeGroup>

</Steps>

## Mapping CSV / JSON to RDF

Most real-world data arrives as CSV or JSON exports, not RDF. The recommended approach is to convert your source data to JSON-LD client-side — mapping each row to your ontology's IRIs and datatypes — then upload the result directly to `POST /graphs/{id}/import`. This keeps the pipeline simple, auditable, and entirely within the public API.

### Convert CSV Rows to JSON-LD

The following Python helper reads a product CSV and produces a JSON-LD document ready for import.

```python Python
import csv
import json

EC = "https://ontologies.anclab.io/ecommerce/v1#"
DATA = "https://data.anclab.io/ecommerce/"

def csv_to_jsonld(csv_path: str) -> dict:
    """Convert a product CSV to a JSON-LD document."""
    context = {
        "ec":    EC,
        "xsd":   "http://www.w3.org/2001/XMLSchema#",
        "hasSKU":    {"@id": f"{EC}hasSKU",   "@type": "xsd:string"},
        "hasName":   {"@id": f"{EC}hasName",  "@type": "xsd:string"},
        "hasPrice":  {"@id": f"{EC}hasPrice", "@type": "xsd:decimal"},
        "belongsTo": {"@id": f"{EC}belongsTo","@type": "@id"},
    }

    graph = []
    with open(csv_path, newline="", encoding="utf-8") as f:
        for row in csv.DictReader(f):
            graph.append({
                "@id":   f"{DATA}product/{row['sku'].lower()}",
                "@type": f"{EC}Product",
                "hasSKU":    row["sku"],
                "hasName":   row["name"],
                "hasPrice":  row["price"],
                "belongsTo": f"{DATA}category/{row['category_slug']}",
            })

    return {"@context": context, "@graph": graph}

jsonld_doc = csv_to_jsonld("new-products.csv")
with open("new-products.jsonld", "w") as f:
    json.dump(jsonld_doc, f, indent=2)

print(f"Converted {len(jsonld_doc['@graph'])} rows to JSON-LD")
```

### Import the JSON-LD into Your Graph

Upload the converted file to `POST /graphs/{id}/import` with the appropriate named graph for provenance tracking.

<CodeGroup>

```bash curl
curl -X POST https://api.anclab.io/v1/graphs/grph_3mNpQrStuv/import \
  -H "Authorization: Bearer anchub_sk_..." \
  -F "file=@new-products.jsonld;type=application/ld+json" \
  -F "namedGraph=https://data.anclab.io/ecommerce/imports/csv-batch-2024-11" \
  -F "validate=true"
```

```python Python
with open("new-products.jsonld", "rb") as f:
    r = requests.post(
        f"{BASE_URL}/graphs/{GRAPH_ID}/import",
        headers={"Authorization": "Bearer anchub_sk_..."},
        files={"file": ("new-products.jsonld", f, "application/ld+json")},
        data={
            "namedGraph": "https://data.anclab.io/ecommerce/imports/csv-batch-2024-11",
            "validate": "true",
        },
    )

job = r.json()
print(f"Import job: {job['jobId']}  status: {job['status']}")
```

</CodeGroup>

The platform validates each generated triple against your SHACL shapes before committing. Rows that violate constraints are reported in the `rejectedRows` field of the completed job response.

<Tip>
To preview your conversion before importing, simply inspect the generated JSON-LD file locally before uploading. Check that each `@id` IRI resolves correctly, `@type` matches your ontology class, and all required properties are present. Fix any issues in your conversion script, then re-run the import.
</Tip>

## Verifying Your Integrations

After loading external data, run a quick audit query to check for triples that arrived without the required properties:

```sparql
PREFIX ec:  <https://ontologies.anclab.io/ecommerce/v1#>
PREFIX owl: <http://www.w3.org/2002/07/owl#>

SELECT ?entity ?type (COUNT(?link) AS ?externalLinks)
WHERE {
  ?entity a ?type .
  OPTIONAL { ?entity owl:sameAs ?external . BIND(1 AS ?link) }
  FILTER(?type IN (ec:Product, ec:Supplier, ec:Category))
}
GROUP BY ?entity ?type
ORDER BY DESC(?externalLinks)
LIMIT 20
```
