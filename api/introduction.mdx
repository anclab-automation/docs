The ANC Lab REST API gives you programmatic access to every feature of the Knowledge Hub platform. Use it to create and manage ontologies, populate and query knowledge graphs, run SPARQL queries, and retrieve graph analytics — all over standard HTTPS using JSON payloads. Every API request targets a versioned base URL so your integrations remain stable as the platform evolves.

## Base URL

All API requests use the following base URL:

```
https://api.anclab.io/v1
```

## API Versioning

The current stable version is **v1**. The version segment is part of the URL path, not a header, so you always know exactly which version your client is calling.

<Note>
  ANC Lab follows a 12-month deprecation policy. When a new major version ships, the previous version continues to receive bug fixes and security patches for at least 12 months. Watch the [changelog](https://docs.anclab.io/changelog) for advance notice of breaking changes.
</Note>

## Content Types

| Scenario | Request `Content-Type` | Response `Content-Type` |
|---|---|---|
| Standard JSON operations | `application/json` | `application/json` |
| Raw SPARQL query body | `application/sparql-query` | `application/json` |
| RDF file import (Turtle) | `text/turtle` | `application/json` |
| RDF file import (multipart) | `multipart/form-data` | `application/json` |

Always set `Accept: application/json` unless you are requesting a specific RDF serialisation from an export endpoint.

## Authentication

Every request must carry your API key in the `Authorization` header:

```
Authorization: Bearer anchub_sk_...
```

See the [Authentication](/api/authentication) page for full details on creating and managing API keys.

## Rate Limits

| Plan | Requests per minute |
|---|---|
| Free | 100 |
| Starter | 500 |
| Pro | 1 000 |
| Enterprise | Custom |

When you exceed the limit the API returns `429 Too Many Requests` with a `Retry-After` header indicating the number of seconds to wait before retrying.

## Pagination

List endpoints support cursor-free, offset-based pagination via two query parameters:

| Parameter | Type | Default | Description |
|---|---|---|---|
| `page` | integer | `1` | The page number (1-indexed) |
| `limit` | integer | `20` | Items per page (max `100`) |

Every paginated response includes a `meta` object:

```json
{
  "meta": {
    "page": 1,
    "limit": 20,
    "total": 143,
    "total_pages": 8
  }
}
```

## Error Codes

The API uses standard HTTP status codes. All error responses share a consistent JSON structure:

```json
{
  "error": {
    "code": "INVALID_INPUT",
    "message": "The field 'namespace_uri' must be a valid absolute IRI.",
    "details": {
      "field": "namespace_uri",
      "provided": "not-an-iri"
    }
  }
}
```

| Status | Meaning | Common cause |
|---|---|---|
| `400 Bad Request` | Invalid input | Missing required field, malformed JSON, or constraint violation |
| `401 Unauthorized` | Missing or invalid API key | No `Authorization` header or an expired/revoked key |
| `403 Forbidden` | Insufficient permissions | Key scope does not allow the requested operation |
| `404 Not Found` | Resource does not exist | Wrong ID, or the resource was deleted |
| `409 Conflict` | State conflict | Duplicate namespace prefix, or concurrent modification |
| `422 Unprocessable Entity` | Semantic validation error | Ontology parse failure, SPARQL syntax error |
| `429 Too Many Requests` | Rate limit exceeded | See `Retry-After` header for backoff duration |
| `500 Internal Server Error` | Platform error | Unexpected server-side failure; contact support with the `request_id` |

<Tip>
  Every response — including errors — includes an `X-Request-Id` header. Quote this value when opening a support ticket so ANC Lab engineers can trace the exact request in internal logs.
</Tip>
