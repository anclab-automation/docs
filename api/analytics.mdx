The Analytics API gives you prebuilt graph metrics — entity distribution, relationship density, centrality, path analysis, and cluster detection — alongside a flexible endpoint for custom SPARQL aggregation queries. Run analyses on demand or schedule recurring reports to be delivered via webhook, so you always have fresh insight into the structure and health of your knowledge graphs.

---

## Run Graph Analytics

```
GET /analytics
```

Executes a built-in analytics metric against a knowledge graph and returns structured results.

### Query Parameters

<ParamField query="graph_id" type="string" required>
  The ID of the knowledge graph to analyse.
</ParamField>

<ParamField query="metric" type="string" required>
  The analytics metric to compute. Accepted values:
  - `entity_distribution` — count of entities grouped by `rdf:type` class
  - `relationship_density` — ratio of triples to entities per predicate
  - `centrality` — PageRank-style importance score for each entity
  - `path_analysis` — shortest-path distances between entity pairs
  - `cluster_detection` — community clusters identified by modularity algorithm
</ParamField>

<ParamField query="entity_class" type="string">
  Filter the metric to entities of a specific class IRI. For example, pass `https://schema.org/Product` to compute centrality only over product entities.
</ParamField>

<ParamField query="page" type="integer">
  Page number for paginated results (1-indexed). Defaults to `1`. Use with `limit` to page through large result sets such as those produced by `cluster_detection` on large graphs.
</ParamField>

<ParamField query="limit" type="integer">
  Maximum number of result rows to return per page. Defaults to `100`, maximum `1000`.
</ParamField>

<Tip>
  For large graphs, combine `entity_class` filtering with a lower `limit` to get fast, focused results. Fetching the top 20 most central entities within a single class is typically an order of magnitude faster than computing centrality across the entire graph.
</Tip>

### Example: Entity Distribution

```bash curl
curl "https://api.anclab.io/v1/analytics?graph_id=grph_03LD5MSRB9YWZP4OFVHJ1TGCK&metric=entity_distribution&limit=10" \
```

## Run a Custom Analytics Query

```
POST /analytics/query
```

Executes an arbitrary SPARQL query with aggregate operations and returns the results in JSON or CSV. Use this endpoint when the built-in metrics do not cover your specific analytical need.

### Request Body

<ParamField body="query" type="string" required>
  A SPARQL 1.1 query that uses aggregate functions (`COUNT`, `SUM`, `AVG`, `MIN`, `MAX`, `GROUP_CONCAT`) and a `GROUP BY` clause.
</ParamField>

<ParamField body="graph_id" type="string" required>
  The ID of the knowledge graph to query.
</ParamField>

<ParamField body="format" type="string">
  Output format: `json` (default) or `csv`.
</ParamField>

### Example Request

```bash curl
curl -X POST https://api.anclab.io/v1/analytics/query \
  -H "Authorization: Bearer anchub_sk_live_abc123XYZ" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "PREFIX schema: <https://schema.org/>\nSELECT ?supplier (COUNT(?product) AS ?productCount) (AVG(?price) AS ?avgPrice)\nWHERE {\n  ?product a schema:Product ;\n           schema:manufacturer ?supplier ;\n           schema:price ?price .\n}\nGROUP BY ?supplier\nORDER BY DESC(?productCount)\nLIMIT 20",
    "graph_id": "grph_03LD5MSRB9YWZP4OFVHJ1TGCK",
    "format": "json"
  }'
```

### Example Response

```json
  "data": {
```

## Create a Saved Report

```
POST /analytics/reports
```

Creates a saved analytics report that you can run on demand or on a recurring schedule.

### Request Body

<ParamField body="name" type="string" required>
  A descriptive name for the report. Example: `"Weekly Entity Distribution"`.
</ParamField>

<ParamField body="graph_id" type="string" required>
  The ID of the knowledge graph to run this report against.
</ParamField>

<ParamField body="metric" type="string" required>
  The analytics metric this report will compute. Same accepted values as `GET /analytics` — `entity_distribution`, `relationship_density`, `centrality`, `path_analysis`, or `cluster_detection`.
</ParamField>

<ParamField body="schedule" type="string">
  A cron expression defining when the report runs automatically. Omit to create a manual-only report. Example: `"0 8 * * 1"` runs every Monday at 08:00 UTC.
</ParamField>

<ParamField body="webhook_url" type="string">
  An HTTPS URL to call with the report results after each scheduled execution. The platform sends a `POST` request with a JSON body containing the report ID and a signed download URL for the results.
</ParamField>

### Example Request

<CodeGroup>

```bash curl
curl -X POST https://api.anclab.io/v1/analytics/reports \
  -H "Authorization: Bearer anchub_sk_live_abc123XYZ" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Weekly Entity Distribution",
    "graph_id": "grph_03LD5MSRB9YWZP4OFVHJ1TGCK",
    "metric": "entity_distribution",
    "schedule": "0 8 * * 1",
    "webhook_url": "https://hooks.myapp.io/anclab-reports"
  }'
```

```python Python
import os, requests

payload = {
    "name": "Weekly Entity Distribution",
    "graph_id": "grph_03LD5MSRB9YWZP4OFVHJ1TGCK",
    "metric": "entity_distribution",
    "schedule": "0 8 * * 1",
    "webhook_url": "https://hooks.myapp.io/anclab-reports",
}

resp = requests.post(
    "https://api.anclab.io/v1/analytics/reports",
    json=payload,
    headers={"Authorization": f"Bearer {os.environ['ANCHUB_API_KEY']}"},
)
resp.raise_for_status()
print("Report created:", resp.json()["data"]["id"])
```

</CodeGroup>

### Example Response

```json
{
  "data": {
    "id": "rpt_04ME6NTSCA1ZPQ5PGWIK2UHDO",
    "name": "Weekly Entity Distribution",
    "graph_id": "grph_03LD5MSRB9YWZP4OFVHJ1TGCK",

    "last_run_status": null,

```

<Note>
  Scheduled reports run in UTC. After each execution, the platform delivers results to your `webhook_url` (if configured) within 60 seconds. The webhook payload contains a `download_url` signed with a 24-hour expiry so your system can fetch the full results at its own pace. Configure retry logic on your webhook receiver — ANC Lab retries failed deliveries up to 3 times with exponential backoff.
</Note>

## Execute a Saved Report

```
GET /analytics/reports/{id}/run
```

Triggers an immediate, on-demand execution of a saved report and returns the results synchronously.

### Path Parameter

<ParamField path="id" type="string" required>
  The unique report ID to execute.
</ParamField>

### Query Parameters

  Override the default result row limit for this execution. Defaults to `100`, maximum `1000`.

## Python Example: Top 10 Central Entities

The following example fetches centrality scores for all products and prints the top 10 by score.

```python Python
import os
import requests

API_KEY  = os.environ["ANCHUB_API_KEY"]
GRAPH_ID = "grph_03LD5MSRB9YWZP4OFVHJ1TGCK"
BASE_URL = "https://api.anclab.io/v1"

headers = {"Authorization": f"Bearer {API_KEY}"}

params = {
    "graph_id":     GRAPH_ID,
    "metric":       "centrality",
    "entity_class": "https://schema.org/Product",
    "limit":        10,
}

resp = requests.get(f"{BASE_URL}/analytics", params=params, headers=headers)
resp.raise_for_status()

results = resp.json()["data"]["results"]

print(f"{'Rank':<6} {'Score':<8} Entity")
print("-" * 70)
for rank, row in enumerate(results, start=1):
    label = row.get("label", row["entity"].split("/")[-1])
    print(f"{rank:<6} {row['score']:<8.4f} {label}")
```

<Tip>
  The `GET /analytics` endpoint returns up to 1 000 rows per request. For metric computations that produce more results — common with `cluster_detection` on large graphs — use the `limit` and `page` parameters together to page through all results without overwhelming your application's memory. Process each page incrementally rather than collecting all rows before analysis.
</Tip>
